> the same text (the May 0.92+ was regime, not regression). The real cost is
> the verify cycle: ~75 ms ctx-flat (3.9‚Äì4.9√ó an MTP=0 token), nsys-decomposed

## Step 1 ‚Äî Two 5-minute A/B probes for the acceptance collapse

Acceptance was 0.92‚Äì0.98 on the same class of text in May (E3). Something
likely regressed it. Two suspects, both testable with existing kill-switches
on the MTP=4 ctx {128, 3072} cells:

1. **`QWEN36_DELTANET_CHUNKED_PREFILL=0`** ‚Äî prime suspect. The verify chunk
   runs DeltaNet through the chunked WY-form prefill kernel
   (`deltanet_prefill.cu`, default ON, engine.rs:5646-5693 picks it for 5-token
   chunks), while the drafts were generated by the sequential decode kernel.
   WY-form is NOT bit-equal to the sequential recurrence ‚Üí borderline argmax
   flips ‚Üí rejected drafts. If tok/s and acceptance jump with this off, the
   fix is: route verify chunks (tokens ‚â§ ~8) to the sequential
   `deltanet_decode` path unconditionally (cheap at 5 tokens), keep chunked
   for real prefill.
2. **`QWEN36_ATTENTION_SAGE_PREFILL=0`** ‚Äî prompts ‚â•1024 are prefilled with
   the INT8 sage kernel but verify chunks run scalar/split: the inverse of the
   "consistency dividend" that gained DFlash +2.7 AL. Cannot explain the
   ctx=128 cell (no sage <1024), so test second.

Also compare with the chat path on the same text (`chat --prompt "$(head -c
12000 benches/data/bench_corpus_91k.txt)" --max-new-tokens 64
--mtp-speculative-tokens 4` + `QWEN36_MTP_STATS=1`) ‚Äî if chat acceptance is
high while bench is low, the regression is in the bench/multi-graph path
specifically (then bisect with `QWEN36_MTP_MULTI_GRAPH_DISABLE=1`).

Gate for this step: a named root cause with an A/B table in DAILY.md.

## Step 2 ‚Äî Structural fix: the verify chunk uses the kernel decode abandoned

Code-proven, independent of step 1, and it is why the gap WIDENS with context:

- Decode got the register-tiled v2 kernel (`attention_decode_tiled.cu`).
- The MTP verify chunk goes through `qwen36_attention_prefill`, where the
  dispatch (`attention.cu:2466-2474`: tokens ‚â§ 8 AND n_splits ‚â• 64) sends it
  to **`attention_prefill_split_gqa_kernel` launched once per token (5√ó)** ‚Äî
  the same serial-per-timestep / 1-byte-loads / 5-syncs-per-position disease
  the decode v1 had (28√ó off bandwidth). **No tiled prefill variant exists.**
- Aggravation: under graph capture n_splits comes from the context bucket
  (min 8192 ‚Üí 128 splits even at ctx 128, mostly empty), and the FA-2
  split-K kernel (which already handles 2‚Äì32-token chunks beautifully for
  DFlash) is intercepted by this split path first AND bails out during
  capture (`cudaStreamIsCapturing` guard).

Fix options, in order of preference:
1. Port the tiled-v2 pattern (8-timestep register tiles, vectorized loads,
   FP8 LUT, batched online softmax) to the split-prefill path ‚Äî same
   partials/reduce contract, multi-token q. Most of the kernel exists in
   `attention_decode_tiled.cu`; the delta is q>1.
2. Or make the verify chunk eligible for `attention_flash_splitk.cu` inside
   capture: the #55 engine-owned-partials work removed the malloc problem;
   what remains is ordering it before the scalar split path for 3‚Äì8-token
   chunks and validating capture-safety.

Gates: smoke parity (cos ‚â• 0.998, BF16+FP8, incl. nonzero start_position and
empty splits), MTP parity floor (hello/hello world √ó MTP {0..4}), dashboard
before/after on MTP=4 @ {3072, 24576}. Expected: the 24K MTP=4 cell stops
being slower than MTP=0; combined with step 1, MTP=4 should beat MTP=0
everywhere acceptance > ~0.4.

## Step 3 ‚Äî Depth 6 (then maybe 8) ‚Äî ONLY after steps 1‚Äì2 land

All locks are verify-buffer constants; the MTP head itself recurses without

| `MTP_MAX_DRAFT_TOKENS` | engine.rs:59 | 6 | 8 |

Caveats:
- **depth 8 ‚Üí verify chunk = 9 tokens > `kPrefillSplitLongChunkMaxTokens=8`**
  (attention.cu) ‚Äî it falls out of the split path entirely. Step 2 must cover
  9-token chunks (the flash split-K already takes 2‚Äì32) or raise that constant.
- Measure **per-position acceptance** (draft 1..6) when instrumenting step 0 ‚Äî
  it decides whether depth 6/8 pays at all. With chain MTP, acceptance
  compounds: position-5/6 drafts only land if all before them did.
- Gate: dashboard MTP=6 ‚â• MTP=4 on ‚â•3 of 4 cells, parity floor extended to
  depth 6.

## Step 4 ‚Äî Routing (the strategic payoff)

Once MTP is healthy at long ctx: auto-route `--drafter` by prompt length ‚Äî
DFlash ‚â§ ~4K (drafter forward is O(n), break-even ~5K today), chain MTP
(depth 6) above; include MTP=0 in the best-of-both (the dashboard proved it
can win). Online fallback: if observed acceptance < ~0.3 after N cycles,
drop to MTP=0. Gate: no dashboard cell below its best single-mode baseline.
